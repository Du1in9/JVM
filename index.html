1. JVM 简介

JVM（Java Virtual Machine），是 java 程序的运行环境（java 二进制字节码的运行环境）。

优点：一次编写，到处运行；自动内存管理，垃圾回收功能；数组下标越界检查；多态。

2.2 虚拟机栈

2.2.1 定义

Java Virtual Machine Stacks （Java 虚拟机栈）

  • 每个线程运行时所需要的内存，称为虚拟机栈

  • 每个栈由多个栈帧（Frame）组成，对应着每次方法调用时所占用的内存

  • 每个线程只能有一个活动栈帧，对应着当前正在执行的那个方法

问题辨析：

  1. 垃圾回收是否涉及栈内存？

    • 垃圾回收主要涉及堆内存，而栈内存的管理通常是由程序的运行时环境自动处理的。

    • 栈内存用于存储局部变量和方法调用信息，不需要垃圾回收。

  2. 栈内存分配越大越好吗？

    • 不一定。栈内存分配过大可能导致栈溢出，而过小则可能限制程序的运行，导致无法处理较深的递归或较大的局部变量。

    • 合适的栈内存大小应根据具体应用需求来调整。

  3. 方法内的局部变量是否线程安全？

    • 如果方法内局部变量没有逃离方法的作用访问，它是线程安全的

    • 如果是局部变量引用了对象，并逃离方法的作用范围，需要考虑线程安全

2.2.2 栈内存溢出

  • 栈帧过多导致栈内存溢出

  • 栈帧过大导致栈内存溢出

2.2.3 线程运行诊断

  • 案例1：CPU 占用过多

    • 用 top 命令定位是哪一个进程对 CPU 的占用过高

    • 用 ps H -eo pid, tid, %cpu | grep 进程id 命令定位是哪个线程引起的

    • 用 jstack 进程id 命令获取线程堆栈信息，再根据 线程id 找到有问题的线程，定位是哪一行代码引起的

  • 案例2：程序运行很长时间没有结果

    • 用 top 命令查看进程状态，确认程序是否仍在运行中（例如，CPU 和内存使用情况）

    • 用 strace -p 进程id 命令监控系统调用，检查程序是否在等待某些资源或调用被阻塞

    • 用 jstack 进程id 命令获取线程堆栈信息，查看各线程的状态，定位是否有线程处于阻塞或死锁状态

2.3 本地方法栈

本地方法是用其他语言（如 C 或 C++）编写的代码，并通过 JNI（Java Native Interface）与 Java 代码交互。

它们通常用于性能优化、访问系统资源或重用已有的库。常见的本地方法：Object.clone()、Thread.start() 等。

本地方法栈与虚拟机栈的区别：

  • 存储内容：Java 栈主要用于存储 Java 方法的局部变量和调用信息，而本地方法栈专门用于本地方法的执行。

  • 语言支持：Java 栈只处理 Java 方法的调用和返回，而本地方法栈支持其他语言编写的方法调用。

2.4 堆

2.4.1 定义

Heap（堆），通过 new 关键字，创建对象都会使用堆内存。

  • 它是线程共享的，堆中对象都需要考虑线程安全的问题

  • 有垃圾回收机制

2.4.2 堆内存溢出

堆内存溢出通常发生在程序分配的内存超出可用堆内存时，导致应用程序崩溃或行为异常。常见于不断分配内存但未释放的循环或递归。

xxxxxxxxxx
public static void main(String[] args) {
    ArrayList<Object> list = new ArrayList<>();
    while (true) {
        list.add(new Object());             // 不断分配内存
    }
}

2.4.3 堆内存诊断

  2. jmap 工具：查看堆内存占用情况 jmap -heap 进程id

  3. jconsole 工具：图形界面的，多功能的监测工具，可以连续监测

2.5 方法区

2.5.1 定义

在 JVM 中，方法区是内存的一部分，用于存储类结构信息，包括类的元数据、静态变量和常量池。

方法区是堆的一部分，但它是专门用于存放类级别的信息，而不是实例级别的。主要作用是为 Java 类提供运行时数据支持。

在 JDK 8 及之后，方法区的实现被称为 metaspace，使用本地内存而非 JVM 堆内存，这有助于解决内存溢出的问题。

2.5.2 组成

2.5.3 方法区内存溢出

  1. Java 1.8 之前：

    方法区通常是一个固定大小的内存区域，可以通过 JVM 启动参数进行设置，如 -XX:PermSize 和 -XX:MaxPermSize。

    如果不断加载大量类，可能会导致方法区内存溢出。

    ​x
    static class OOMObject {}
    

    public static void main(String[] args) {
        List<OOMObject> list = new ArrayList<>();
        while (true) {
            list.add(new OOMObject());              // 持续创建对象，导致方法区不断加载类信息
        }
    }

  2. Java 1.8 之后：

    方法区被移除，取而代之的是 Metaspace。直接使用本地内存，默认没有大小限制，除非通过 -XX:MaxMetaspaceSize 进行设置。

    因此，方法区内存溢出的问题在 Java 1.8 之后主要与本地内存的使用有关。

    xxxxxxxxxx
    static class OOMObject {}
    

    public static void main(String[] args) {
        List<Class<?>> classList = new ArrayList<>();
        while (true) {
            classList.add(OOMObject.class);         // 不断加载类信息，可能导致 Metaspace 溢出
        }
    }

2.5.4 运行时常量池

运行时常量池（Runtime Constant Pool）是方法区的一部分，用于存放编译期间生成的各种字面量和符号引用。

运行时常量池在类被加载时创建，并在类的生命周期内存储常量值，比如字符串、数值、类引用等。

  1. 动态性：在类加载时创建，可以在运行时添加常量（如字符串）。

  2. 共享性：同一个类的多个实例共享常量池中的常量。

  3. 存储类型：包含字符串字面量、数值、方法和字段的符号引用等。

2.5.5 StringTable

StringTable 通常存储在方法区（在 Java 8 之前）或元空间（在 Java 8 之后）。其内存区域负责存储类的元数据以及静态变量。

  • 字符串池：StringTable 是一个哈希表，用于存储所有被创建的字符串字面量和字符串常量。

    当你在代码中直接定义字符串时，JVM 会将这些字符串放入 StringTable 中。

  • 唯一性：通过 StringTable，相同的字符串字面量不会被重复存储，确保内存的有效利用。

    比如，两个相同的字符串字面量会指向同一个内存地址。

  • 快速比较：由于字符串是常量池中的引用，使用 == 比较字符串时可以快速判断引用是否相同。

xxxxxxxxxx
String s1 = "a";                                    // 常量池中的 "a"
String s2 = "b";                                    // 常量池中的 "b"
String s3 = "a" + "b";                              // 编译时常量，常量池中的 "ab"
String s4 = s1 + s2;                                // 运行时计算，s4 是一个新的对象
String s5 = "ab";                                   // 常量池中的 "ab"
String s6 = s4.intern();                            // 将 s4 的值添加到常量池，s6 指向常量池中的 "ab"


System.out.println(s3 == s4);                       // false
System.out.println(s3 == s5);                       // true
System.out.println(s3 == s6);                       // true
xxxxxxxxxx
String x2 = new String("c") + new String("d");      // 创建新的字符串对象，x2 指向新内存
String x1 = "cd";                                   // 常量池中的 "cd"
x2.intern();                                        // 将 x2 的内容添加到常量池
System.out.println(x1 == x2);                       // false


// 如果调换前两行代码的位置，也不会影响结果。即使 x2 调用了 intern()，x1 仍然指向常量池中的 "cd"，而 x2 是一个新对象。

  1. StringTable 垃圾回收

    • 垃圾回收机制：一般来说，StringTable 中的字符串常量不会被垃圾回收，因为它们在类加载时被加载，

      并在整个程序运行期间存在。只有在类卸载时，相关的字符串常量才会被回收。

    • 调整大小：当 StringTable 的大小超过某个阈值时，JVM 会进行扩展，增加其容量。这样可以减少哈希冲突，提高查找效率。

  2. StringTable 性能调优

    • 初始容量：可以通过 JVM 参数 -XX:InitialStringTableSize 来设置 StringTable 的初始大小，以避免在运行时多次扩展。

    • 负载因子：StringTable 的负载因子会影响查找性能。如果知道将使用大量字符串，可以适当增加初始大小以降低扩展频率。

    • 使用 String.intern()：将字符串显式地添加到 StringTable 中，从而避免在运行时多次创建相同字符串对象。

2.6 直接内存

2.6.1 定义

直接内存（Direct Memory）是指不在 JVM 堆中分配的内存，常用于高性能的 NIO 操作。

虽然分配和回收直接内存的成本较高，但它提供了更高的读写性能，因为可以绕过 JVM 的内存管理机制。

xxxxxxxxxx
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);            // 分配直接内存


buffer.put("Hello, Direct Memory!".getBytes());                 // 写入数据


buffer.flip();                                                  // 切换到读模式
byte[] data = new byte[buffer.remaining()];
buffer.get(data);                                               // 读取数据


System.out.println(new String(data));

2.6.2 分配和回收原理

直接内存的分配和回收依赖于 Unsafe 类，这是一种低级别的操作方式。

分配时使用 Unsafe 直接分配内存，而释放则需要主动调用 freeMemory 方法。

此外，ByteBuffer 的实现通过 Cleaner（虚引用）监测其对象的生命周期，

当 ByteBuffer 被垃圾回收时，ReferenceHandler 线程会调用 Cleaner 的 clean 方法来释放直接内存。

xxxxxxxxxx
ByteBuffer buffer = ByteBuffer.allocateDirect(1024);            // 分配直接内存


Cleaner cleaner = ((DirectByteBuffer) buffer).cleaner();        // 手动获取 Cleaner


// ... (使用 buffer)


cleaner.clean();                                                // 触发垃圾回收并释放内存

 

 

三、垃圾回收

3.1 如何判断对象可以回收

3.1.1 引用计数法

引用计数法是一种内存管理机制，通过对对象的引用次数进行计数来判断其是否可以被回收。

当对象的引用计数为零时，说明没有任何引用指向该对象，可以安全地回收。

xxxxxxxxxx
class Node {
    int value;
    int refCount = 0;


    Node(int value) {
        this.value = value;
    }


    void addReference() {
        refCount++;
    }


    void removeReference() {
        refCount--;
        if (refCount == 0) {
            // 这里可以添加释放资源的逻辑
        }
    }
}
xxxxxxxxxx
Node node = new Node(10);
node.addReference();            // 引用计数加1
node.removeReference();         // 引用计数减1，变为0，准备回收

3.1.2 可达性分析算法

垃圾回收器使用可达性分析来管理内存。这种方法从一组称为 GC Roots 的对象开始扫描，

通过引用链判断哪些对象是可达的（使用中的），哪些是不可达的（可以回收的）。GC Root 对象包括：

  1. 虚拟机栈中的引用：每个线程的栈帧中存储的局部变量。

  2. 方法区中的静态引用：类的静态字段或常量。

  3. JNI（Java Native Interface）引用：通过本地方法持有的对象引用。

  4. 活动线程：当前正在运行的线程。

3.1.3 四种引用

1. 强引用（Strong Reference）

强引用是 Java 中最常见的引用类型。如果一个对象有强引用指向它，那么即使系统内存紧张，垃圾回收器也不会回收这个对象。

只有当所有的 GC Roots 不再强引用该对象时，该对象才会被回收。

xxxxxxxxxx
Object strongReference = new Object();                                                      // 强引用

2. 软引用（SoftReference）

软引用用于缓存，只有在内存不足时，垃圾回收器才会回收被软引用指向的对象。使用软引用可以减少内存占用。

xxxxxxxxxx
SoftReference<Object> softRef = new SoftReference<>(new Object());                          // 软引用

3. 弱引用（WeakReference）

弱引用的对象在垃圾回收时，无论内存是否充足，都会被回收。弱引用常用于实现规范化的缓存。

xxxxxxxxxx
WeakReference<Object> weakRef = new WeakReference<>(new Object());                          // 弱引用

4. 虚引用（PhantomReference）

虚引用无法单独使用，必须与引用队列结合。它主要用于清理和管理直接内存（如 ByteBuffer）。

在对象被回收后，虚引用会被加入到引用队列中，允许程序在对象被回收后执行某些清理操作。

xxxxxxxxxx
ReferenceQueue<Object> refQueue = new ReferenceQueue<>();
PhantomReference<Object> phantomRef = new PhantomReference<>(new Object(), refQueue);       // 虚引用

5. 终结器引用（FinalReference）

终结器引用用于对象的终结过程，垃圾回收器会在回收对象之前将其放入引用队列，并通过终结器线程调用其 finalize 方法。

这个过程是隐式的，程序员通常不需要手动管理。Java 中没有直接创建终结器引用的方式，通常使用 finalize 方法来展示其用法。

xxxxxxxxxx
class FinalizeExample {
    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        System.out.println("Finalize method called");
    }
}
xxxxxxxxxx
FinalizeExample example = new FinalizeExample();
example = null;                                     // 将引用置为 null，便于垃圾回收
System.gc();                                        // 强制进行垃圾回收

3.2 垃圾回收算法

3.2.1 标记清除

  1. 标记阶段：遍历所有可达对象，从根对象开始，标记所有可达的对象。

  2. 清除阶段：遍历堆内存，回收未被标记的对象。

  • 优点：实现简单，直观。缺点：可能导致内存碎片。

3.2.2 标记整理

  1. 标记阶段：与标记清除法相同，标记所有可达对象。

  2. 整理阶段：移动所有可达对象到堆的起始位置，并更新引用。

  • 优点：消除内存碎片。缺点：移动对象可能带来额外的开销。

3.2.3 复制

  1. 将堆分为两块（通常是 “年轻代” ）。

  2. 活动对象从一个块复制到另一个块，未使用的块则被清空。

  • 优点：高效，适合大量短生命周期的对象。缺点：需要额外的内存空间。

3.3 分代垃圾回收

  1. 对象分配：新创建的对象通常分配在新生代的伊甸园区域。

  2. Minor GC：当伊甸园区域内存不足时，触发 Minor GC。此时，伊甸园和 “from” 空间中的存活对象会被复制到 “to” 空间。

    存活对象的年龄会增加 1，并且 “from” 和 “to” 空间会交换角色。

  3. Stop the World (STW)：Minor GC 会导致所有用户线程暂停，直到垃圾回收完成后才能继续执行。

  4. 对象晋升：如果对象的存活时间超过设定阈值，它会被晋升到老年代，最大年龄是15（用 4 位表示）。

  5. Full GC：当老年代空间不足时，首先尝试 Minor GC，如果仍然不够，再触发 “Full GC”，这时 STW 时间会更长。

xxxxxxxxxx
// 1. 模拟对象分配
for (int i = 0; i < 10000; i++) {
    Object obj = new Object();          // 对象分配在伊甸园区域
}
// 2. 触发 Minor GC 的方式（简化版）
System.gc();                            // 请求垃圾回收


// 3. JVM 会在此处尝试进行 Minor GC

3.4 垃圾回收器

JVM 在垃圾回收策略上有三种不同模式：

  1. 串行：采用单线程进行垃圾回收，适合堆内存较小的环境，通常用于个人电脑，简单高效，但不适合大规模应用。

  2. 吞吐量优先：通过多线程来优化性能，适合堆内存较大且支持多核 CPU 的系统。

    目标是降低STW（Stop The World）时间，使得单位时间内的有效处理时间最大化，从而提高整体吞吐量。

  3. 响应时间优先：同样使用多线程，尽量缩短每次 STW 的时间，以提高系统对请求的响应速度，适用于对实时性要求高的应用

3.4.1 串行

要启用串行垃圾回收器，运行时加上 -XX:+UseSerialGC 参数即可。

3.4.2 吞吐量优先

  1. -XX:+UseParallelGC：启用并行垃圾回收，适用于年轻代。

  2. -XX:+UseParallelOldGC：启用并行老年代垃圾回收（可选）。

  3. -XX:+UseAdaptiveSizePolicy：自动调整堆内存大小以优化性能。

  4. -XX:GCTimeRatio=ratio：设置垃圾回收时间与应用时间的比例，决定GC的频率。

  5. -XX:MaxGCPauseMillis=ms：设置最大垃圾回收暂停时间。

  6. -XX:ParallelGCThreads=n：指定并行垃圾回收线程的数量。

xxxxxxxxxx
java -XX:+UseParallelGC -XX:+UseAdaptiveSizePolicy -XX:GCTimeRatio=4 -XX:MaxGCPauseMillis=50 -XX:ParallelGCThreads=4 ParallelGCExample

3.4.3 响应时间优先

  1. -XX:+UseConcMarkSweepGC：启用并发标记清除垃圾回收器，适用于需要低延迟的应用。

  2. -XX:+UseParNewGC：在年轻代使用并行收集器，适用于多线程环境。

  3. -XX:+UseSerialOld：在老年代使用串行收集器（可选）。

  4. -XX:ParallelGCThreads=n：设置并行垃圾回收的线程数。

  5. -XX:ConcGCThreads=threads：设置用于CMS的并发线程数。

  6. -XX:CMSInitiatingOccupancyFraction=percent：设置触发CMS的老年代占用百分比。

  7. -XX:+CMSScavengeBeforeRemark：在执行标记操作前进行年轻代收集，以减少停顿时间。

xxxxxxxxxx
java -XX:+UseConcMarkSweepGC -XX:+UseParNewGC -XX:ParallelGCThreads=4 -XX:ConcGCThreads=2 -XX:CMSInitiatingOccupancyFraction=75 -XX:+CMSScavengeBeforeRemark CMSExample

3.4.4 G1

Garbage First (G1) 是一种新的垃圾回收器。

2004 年首次提出论文，2009 年在 JDK 6u14 中初步体验，2012 年在 JDK 7u4 中正式支持，2017 年成为 JDK 9 的默认垃圾回收器。

  • G1 特别适用于需要同时兼顾 吞吐量 和 低延迟 的应用。它的默认暂停时间目标为 200 毫秒，适合内存较大的应用场景

  • G1 将堆内存划分为多个相同大小的区域，其垃圾回收策略采用标记+整理算法，而在不同区域之间使用复制算法来优化内存回收

    • -XX:+UseG1GC：启用 G1 垃圾回收器。

    • -XX:G1HeapRegionSize=size：设置 G1 堆区域的大小。

    • -XX:MaxGCPauseMillis=time：设置最大垃圾回收暂停时间。

  1. G1 垃圾回收阶段

    • 年轻代回收（Young Collection）：主要针对新创建的对象，会造成短暂的暂停。回收过程包括：

      • 初始标记：标记年轻代中的存活对象。

      • 复制：将存活对象复制到其他存活区域，释放未被标记的空间。

  • 混合回收（Mixed Collection）：同时回收年轻代和部分老年代。相比年轻代回收，暂停时间通常较长。

    • 选择一些老年代区域进行回收，标记存活对象。

    • 通过回收不同年龄的对象，尽量优化内存使用。

  • 年轻代回收 + 并发标记（Young Collection + Concurrent Mark）：这一阶段结合了年轻代的回收与并发标记。

    • 在进行年轻代的收集时，G1会并发进行标记，找到存活的对象。

    • 通过这种方式，减少了应用的暂停时间，同时优化了内存管理。

  1. Full GC

    不同 JVM 垃圾回收器（GC）在处理新生代和老年代内存不足时有不同的行为。

    • SerialGC

      • 新生代内存不足：发生minor GC，仅清理新生代，通常使用单线程进行回收。

      • 老年代内存不足：发生full GC，会清理整个堆内存，包括新生代和老年代。

    • ParallelGC

      • 新生代内存不足：发生minor GC，使用多个线程并行回收新生代。

      • 老年代内存不足：发生full GC，同样会回收整个堆内存。

    • CMS（Concurrent Mark-Sweep）

      • 新生代内存不足：发生minor GC，使用并发方式回收新生代。

      • 老年代内存不足：发生full GC，清理整个堆内存，但其优化重点在于减少停顿时间。

    • G1（Garbage First）

      • 新生代内存不足：发生minor GC，并发回收新生代。

      • 老年代内存不足：发生full GC，同样会回收整个堆内存。

  2. Young Collection 跨代引用

    • 新生代与老年代：

      • 新生代：存放新创建对象的内存区域，垃圾回收频繁。

      • 老年代：存放经过多次垃圾回收后仍然存活的对象，垃圾回收相对较少。

      当老年代的对象引用了新生代的对象时，就产生了跨代引用。可能导致新生代的对象在老年代的对象存在时仍然无法被回收。

    • 卡表（Card Table）与 Remembered Set：

      • 卡表：用于记录哪些新生代的对象可能被老年代对象引用。每个卡片对应一定的内存区域，用于追踪引用的变化

      • Remembered Set：指所有可能引用新生代对象的老年代对象的集合。在每次垃圾回收时更新，以确保新生代对象正确回收

      Post-write Barrier：在对象引用发生变化时，使用后写障碍（post-write barrier）来更新卡表，确保引用的变化被记录。

      Dirty Card Queue：用于存储那些发生了引用变更的卡片信息，在后续的垃圾回收过程中使用。

      Concurrent Refinement Threads：负责在垃圾回收时更新 Remembered Set，以便垃圾回收器能了解哪些对象仍然存活。

  3. Remark

    • Pre-write Barrier：这是一个操作，在对象的字段被写入之前插入的检查机制。

      它的主要作用是确保在更新对象的引用时，任何相关的状态（例如，对象的可达性）都能得到妥善管理。

      这有助于避免在并发环境中因对象状态不一致而引发的问题。

    • SATB Mark Queue：SATB（Snapshot at the Beginning），是一种标记机制。

      用于在并发垃圾回收期间记录在垃圾收集开始时对象状态。SATB Mark Queue 是一个队列，存储此状态下被修改的对象引用。

      这使得 GC 可以在进行标记和清理操作时，确保不会遗漏任何在垃圾回收开始后仍然活跃的对象。

  4. JDK 8u20 字符串去重

    • 字符串去重：指的是在 JVM 中，如果发现多个字符串内容相同，就让它们共享同一个字符数组，从而减少内存占用。

      • 优点：节省内存，尤其在内存使用较大的情况下，可以显著降低内存的总占用。

      • 缺点：去重过程需要在新生代回收时进行额外的检查，会略微增加 CPU 的使用时间，并且新生代的回收时间可能会稍长。

      String s1 = new String("hello");
      String s2 = new String("hello");
      

      // 在这里假设垃圾回收已经发生并进行了字符串去重
      System.out.println("s1 and s2 are the same object: " + (s1.toCharArray() == s2.toCharArray()));
      

      // 输出内容，验证它们内容相同
      System.out.println(s1.equals(s2));              // true

      String.intern() 是通过字符串常量池来优化字符串的存储，而字符串去重是动态的，主要关注内存中的字符数组。

  5. JDK 8u40 并发标记类卸载

    • JDK 9 引入了更高效的垃圾回收机制，包含了250+ 增强和 180+ bug 修复。这些改进使得内存管理更加高效。

    • JDK 9 及之后的版本中，许多垃圾收集器（如 G1 和 Parallel GC）进行了优化，以更好地适应现代应用程序的需求。

3.5 垃圾回收调优

3.5.1 调优领域

  • 内存：合理设置堆大小、选择合适的垃圾收集器。

  • 锁竞争：优化线程同步以减少阻塞和提高并发性。

  • CPU占用：监控和优化垃圾收集的时间，以减少应用程序的停顿时间。

  • I/O：优化输入输出操作，确保在垃圾收集时不会造成额外的性能损失。

3.5.2 确定目标

  • 低延迟：适合对响应时间要求严格的应用，比如实时系统。

    可以选择 CMS（Concurrent Mark-Sweep）、G1（Garbage-First）或 ZGC（Z Garbage Collector）。

  • 高吞吐量：适合处理大量数据的后台应用。可以选择 Parallel GC（并行垃圾收集器）或 Zing（通过适应性优化实现高吞吐量）。

xxxxxxxxxx
java -XX:+UseG1GC -jar YourApplication.jar                      # 低延迟（G1）
xxxxxxxxxx
java -XX:+UseParallelGC -jar YourApplication.jar                # 高吞吐量（Parallel GC）
xxxxxxxxxx
java -XX:+UseZGC -jar YourApplication.jar                       # 低延迟（ZGC）

3.5.3 最快的 GC

垃圾回收调优的目标是尽量减少 GC 的发生，理想情况下是完全不发生。通过查看 Full GC 前后的内存占用，可以分析以下几个问题：

  1. 数据量是否过多：查询的数据集是否超出了合理范围，导致内存压力增大。

  2. 数据表示是否臃肿：检查对象图和对象大小，以确认对象的内存使用是否优化。例如，多个 Integer 对象可能占用过多内存。

  3. 是否存在内存泄漏：使用静态集合（如static Map）可能导致无法回收的对象，增加内存占用。

3.5.4 新生代调优

新生代中，对象创建成本低，绝大多数对象在短时间内死亡，因此 Minor GC 的开销远小于 Full GC。

合理配置新生代的大小非常关键，过小会导致频繁的 Minor GC，过大则可能导致长时间的 Full GC。主要调优参数包括：

  • -Xmn：设置新生代的初始和最大大小。

  • -XX:MaxTenuringThreshold：调整对象晋升到老年代的阈值。

3.5.5 老年代调优

老年代的垃圾回收调优，尤其以 CMS（Concurrent Mark-Sweep）垃圾收集器为例。主要要点包括：

  • 老年代内存配置：CMS 的老年代越大，能容纳更多的长期存活对象，从而减少 Full GC 的频率。

  • 调优流程：在进行调优时，首先可以尝试不做任何配置。如果没有发生 Full GC，那么内存配置可能已经足够。

    如果发生了 Full GC，就需要观察老年代的内存占用，并考虑将其预设增大约 25% 到 33%。

  • 调优参数：使用 -XX:CMSInitiatingOccupancyFraction 参数可以设置触发 CMS 的阈值。

四、类加载与字节码技术

4.1 类文件结构

JVM 的类文件结构是 Java 字节码的基础，定义了 .class 文件的格式。下面是类文件的主要结构及其组成部分的介绍：

  1. 魔数 (Magic Number)：长度：4 字节。

    • 内容：固定值 0xCAFEBABE。作用：用于标识该文件是一个有效的 Java 类文件。

  2. 版本号 (Version)：长度：4 字节。

    • 内容：主版本号和次版本号。作用：指示编译该类文件所使用的 Java 版本。

  3. 常量池 (Constant Pool)：长度：可变。

    • 内容：存储类中的常量，包括字符串、类等。作用：提供对类中使用的常量的快速访问。

  4. 访问标志 (Access Flags)：长度：2 字节。

    • 内容：描述类的修饰符（如 public、final等）。作用：指示类的可见性和特性。

  5. 类索引 (This Class)：长度：2 字节。

    • 内容：指向常量池中该类的索引。作用：标识当前类。

  6. 父类索引 (Super Class)：长度：2 字节。

    • 内容：指向常量池中父类的索引。作用：指示当前类的父类。

  7. 接口索引 (Interfaces)：长度：2 字节。

    • 内容：指向常量池中实现的接口的索引。作用：列出当前类实现的接口。

  8. 字段表 (Fields)：长度：可变。

    • 内容：描述类中的字段（属性）。作用：定义类的属性及其修饰符。

  9. 方法表 (Methods)：长度：可变。

    • 内容：描述类中的方法。作用：定义类的方法及其实现。

  10. 属性表 (Attributes)：长度：可变。

    • 内容：存储类的附加信息，如源文件名、行号表等。作用：提供额外的元数据。

4.2.6 多态的原理

多态是面向对象编程的一个重要特性，允许不同类的对象对同一消息做出响应。多态的实现主要依赖于继承和方法重写。

Java 中的多态通常分为两种类型：编译时多态（静态多态）和运行时多态（动态多态）。通常所说的多态指的都是运行时多态。

  1. 多态的实现原理

    • 动态绑定与静态绑定：

      • 静态绑定：在编译时确定方法的调用，例如静态方法和构造方法。

      • 动态绑定：在运行时根据对象的实际类型确定调用的方法，这通常涉及到实例方法的调用。

    • 方法调用指令：

      • invokestatic：用于调用静态方法。

      • invokespecial：用于调用构造方法、私有方法和父类方法。

      • invokevirtual：用于调用实例方法，支持多态。

      • invokeinterface：用于调用接口方法，运行时确定具体实现。

      • invokedynamic：用于动态解析方法。

    • 虚方法表：

      • 每个类都有一个虚方法表（vtable），用于存储该类的实例方法的直接引用。

      • 当调用一个方法时，JVM 会根据对象的实际类型查找相应的虚方法表，从而确定调用哪个具体的方法。

        }
    }

        public void eat() {
            System.out.println("Cat eats fish.");
        }

4.2.7 异常处理

在 Java 中，异常处理是一种结构，用于处理程序执行中发生的异常情况。

JVM 提供了一套机制来处理异常，包括 try-catch 语句、finally 子句以及异常表（Exception Table）。

  1. try-catch：用于捕获并处理可能发生的异常。一个 try 块后面可以跟随一个或多个 catch 块，以及一个可选的 finally 块。

  2. finally：无论是否捕获到异常，finally 块中的代码都会执行。通常用于释放资源，如关闭文件流或数据库连接。

    • 多个 single-catch 块的情况：允许根据不同的异常类型来捕获异常。每个 catch 块负责处理一种特定的异常。

    xxxxxxxxxx
    try {
        // 可能会抛出多种异常
    } catch (IOException e) {
        // 处理 IOException
    } catch (NullPointerException e) {
        // 处理 NullPointerException
    }

    • multi-catch 的情况：Java 7 引入，允许一个 catch 块捕获多个异常类型。使得异常处理更加简洁。

    xxxxxxxxxx
    try {
        // 可能会抛出多种异常
    } catch (IOException | SQLException e) {
        // 同时处理 IOException 和 SQLException
    }

  3. 面试题

    • 如果在 finally 块中修改了局部变量，那么这个修改会影响到 try 块中返回的值。

    • 如果 finally 块中执行了 return 语句，那么它会覆盖 try 块或 catch 块中的 return 语句。

    xxxxxxxxxx
    public static void main(String[] args) {
        int result = test();
        System.out.println("Result: " + result);        // Result: 2
    }
    

    static int test() {
        int value = 0;
        try {
            value = 1;
            return value;
        } finally {
            value = 2;
            return value;                               // 这个 return 会覆盖 try 块中的 return
        }
    }

4.2.8 synchronized

synchronized在 JVM 中通过 Monitor 机制实现同步，确保多个线程在访问共享资源时的一致性和线程安全。

通过字节码指令 monitorenter和monitorexit管理同步代码块的锁，通过访问标志 ACC_SYNCHRONIZED 管理同步方法的锁。

随着竞争的增加，锁可以从偏向锁升级到轻量级锁，再到重量级锁，以适应不同的同步需求。

  1. synchronized 修饰代码块

    在字节码层面，下述代码当线程执行到 monitorenter 指令时，会尝试获取锁对象：

    • 如果成功则继续执行同步块内的代码，执行完成后在monitorexit指令处释放锁对象。

    • 如果获取锁失败，则线程会阻塞直到锁对象可用。

    xxxxxxxxxx
    public void syncTask() {
        synchronized(this) {
            // 需要同步的代码块
        }
    }

  2. synchronized 修饰方法

    通过在方法的访问标志中添加 ACC_SYNCHRONIZED 来实现。这意味着当线程调用这个方法时，它必须先获取类实例的锁。

    在字节码层面，JVM 会检查该方法是否有 ACC_SYNCHRONIZED 标志，如果有，则在方法执行期间持有锁对象。

    xxxxxxxxxx
    public synchronized void syncMethod() {
        // 方法体
    }

4.4.1 加载

  • 将类的字节码载入方法区中，内部采用 C++ 的 instanceKlass 描述 java 类，它的重要 field 有：

4.5.3 双亲委派模式

双亲委派模式是 JVM 类加载机制的一种设计原则，旨在确保 Java 类的安全性和一致性。

每个类加载器在加载类时，首先将加载请求委派给它的父类加载器，只有在父加载器无法加载该类时，子加载器才会尝试加载。

  • 原理

    1. 安全性：可以防止核心 Java 类被重复加载。

    2. 一致性：确保同一类在 JVM 中只有一个副本。

  • 流程

    1. 当请求加载一个类时，子加载器先将请求委派给父加载器。

    2. 如果父加载器能加载该类，则返回该类的 Class 对象；如果不能加载，子加载器再尝试加载。

  • 例子

    xxxxxxxxxx
    class MyClassLoader extends ClassLoader {
        @Override
        protected Class<?> findClass(String name) throws ClassNotFoundException {
            // 这里可以自定义类的加载逻辑
            System.out.println("Custom ClassLoader: " + name);
            return super.findClass(name);       // 委派给父类加载器
        }
    }
    xxxxxxxxxx
    MyClassLoader myClassLoader = new MyClassLoader();
    // 1. 尝试加载一个 Java 核心类（如 String），此时被委派并由父加载器处理
    Class<?> stringClass = myClassLoader.loadClass("java.lang.String");
    System.out.println("Loaded class: " + stringClass);
    

    // 2. 尝试加载一个自定义类（如 Test），此时由 MyClassLoader 自行加载
    Class<?> customClass = myClassLoader.loadClass("Test");
    System.out.println("Loaded class: " + customClass);

4.6 运行期优化

4.6.1 即时编译

即时编译（Just-In-Time Compilation）是 JVM 中提高程序执行效率的技术。JIT 编译器会在程序运行时将字节码编译成机器码。

  1. 分层编译：将编译过程分为不同的层次，每个层次对应不同的优化级别。例如，HotSpot JVM 中的分层编译分为以下几个层次：

    • 第 0 层（C0）：不进行任何优化，快速编译。

    • 第 1 层（C1）：进行一些基本优化。

    • 第 2 层（C2）：进行更深入的优化。

    xxxxxxxxxx
    // 示例代码：一个简单的循环
    public static void main(String[] args) {
        for (int i = 0; i < 10000; i++) {
            // ...
        }
    }

    在执行时，JVM 可能会先用 C0 编译，如果发现这段代码被频繁调用，那么可能会使用 C1 或 C2 重新编译以获得更好的性能。

  2. 方法内联：JIT 编译器优化代码的一种方式，它将一个方法的代码直接插入到调用该方法的地方，减少方法调用的开销。

    xxxxxxxxxx
    public static int add(int a, int b) {
        return a + b;
    }
    

    public static void main(String[] args) {
        int result = add(1, 2);
    }

    在JIT编译时，add 方法可能会被内联到 main 方法中，这样就不需要进行方法调用的额外开销。

  3. 字段优化：JIT 编译器在运行时对类的字段进行优化，例如消除不必要的字段访问，或者将字段的访问转化为直接访问。

    xxxxxxxxxx
    public class MyClass {
        private int value;
    

        public int getValue() {
            return value;
        }
    

        public void setValue(int value) {
            this.value = value;
        }
    }

    如果 MyClass 的实例被频繁访问，JIT编译器可能会优化 getValue 和 setValue 方法，使得对 value 字段的访问更加高效。

4.6.2 反射优化

反射是 Java 提供的一种强大特性，允许程序在运行时动态地访问和操作类、接口、字段和方法。

但是反射操作通常比直接代码调用要慢，因此 JVM 提供了一些优化措施。

  1. 缓存：JVM 会缓存反射调用的结果，例如，当使用 Class.getMethod 获取方法时，

    JVM 会将方法的引用缓存起来，以便后续调用时直接使用，而不需要再次进行查找。

    xxxxxxxxxx
    Method method = MyClass.class.getMethod("myMethod", String.class);
    

    // 在这个例子中，第一次调用 `getMethod` 后，JVM会缓存 `myMethod` 的引用，后续调用将直接返回缓存的结果。

  2. 动态代理：反射的另一种优化方式，它允许在运行时动态地创建代理类，

    这些代理类实现了一组接口，并且可以将方法调用转发到指定的调用处理器。

    xxxxxxxxxx
    public interface MyInterface {
        void myMethod();
    }
    

    public class MyInvocationHandler implements InvocationHandler {
        @Override
        public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
            // 处理方法调用
            return null;
        }
    }
    

    MyInterface proxyInstance = (MyInterface) Proxy.newProxyInstance(
        MyInterface.class.getClassLoader(),
        new Class<?>[] { MyInterface.class },
        new MyInvocationHandler()
    );
    

    // 在这个例子中，`MyInvocationHandler` 会在运行时动态地处理 `MyInterface` 接口的方法调用。

  3. 类型检查：反射操作通常涉及类型检查，JVM 会优化这些检查，例如，通过缓存常量池中的类型信息来减少检查的开销。

 

 

五、内存模型

5.1 Java 内存模型

5.1.1 原子性

原子性是指一个操作或者多个操作作为一个不可分割的单元被执行，要么全部执行，要么全部不执行。

在 Java 内存模型中，原子性操作保证了在多线程环境下对共享变量的访问不会出现中间状态。

xxxxxxxxxx
public class Counter {
    private int count = 0;


    public synchronized void increment() {
        count++;
    }


    public synchronized int getCount() {
        return count;
    }
}

increment 方法是原子性的，因为它被 synchronized 关键字修饰，确保了在同一时刻只有一个线程可以执行该方法。

5.1.2 问题分析

Java 内存模型中的问题通常与 多线程 环境下的 共享变量 访问有关，主要问题包括：

  1. 可见性问题：一个线程修改了共享变量的值，其他线程可能看不到这个修改。

  2. 原子性问题：非原子性的操作可能会被中断，导致中间状态的出现。

  3. 指令重排问题：为了优化性能，编译器和处理器可能会改变指令的执行顺序。

xxxxxxxxxx
public class VisibilityDemo {
    private boolean flag = false;
    private int number = 0;


    public void readerThread() {
        while (!flag) {
            Thread.yield();         // 让出 CPU 时间片
        }
        System.out.println(number);
    }


    public void writerThread() {
        number = 42;                // 写操作
        flag = true;                // 标记写操作完成
    }
}

如果 readerThread 在 flag 被设置为 true 之前检查了 flag，那么可能会读取到一个未初始化的 number。

5.1.3 解决方法

  1. 使用 volatile 关键字：确保对共享变量的读写操作对所有线程都是可见的。

    xxxxxxxxxx
    public class VisibilityDemo {
        private volatile boolean flag = false;
        private int number = 0;
    

        public void readerThread() {
            while (!flag) {
                Thread.yield();         // 让出CPU时间片
            }
            System.out.println(number);
        }
    

        public void writerThread() {
            number = 42;                // 写操作
            flag = true;                // 标记写操作完成
        }
    }

  2. 使用 synchronized 关键字：确保同一时刻只有一个线程可以执行某个代码块。

    xxxxxxxxxx
    public class Counter {
        private int count = 0;
    

        public synchronized void increment() {
            count++;
        }
    

        public synchronized int getCount() {
            return count;
        }
    }

  3. 使用锁：例如 ReentrantLock，提供比 synchronized 更灵活的锁定机制。

    xxxxxxxxxx
    public class Counter {
        private final Lock lock = new ReentrantLock();
        private int count = 0;
    

        public void increment() {
            lock.lock();
            try count++;
            finally lock.unlock();
        }
    

        public int getCount() {
            lock.lock();
            try return count; 
            finally lock.unlock();
        }
    }

  4. 使用原子类：例如 AtomicInteger，提供原子操作。

    xxxxxxxxxx
    public class Counter {
        private AtomicInteger count = new AtomicInteger(0);
    

        public void increment() {
            count.incrementAndGet();
        }
    

        public int getCount() {
            return count.get();
        }
    }

  5. 禁止指令重排：使用 volatile 关键字或者锁来禁止指令重排。

5.2 可见性

5.2.1 退不出的循环

先来看一个现象，main 线程对 run 变量的修改对于 t 线程不可见，导致了 t 线程无法停止：

xxxxxxxxxx
static boolean run = true;


public static void main(String[] args) throws InterruptedException {
    Thread t = new Thread(() -> while(run) {...});
    t.start();


    Thread.sleep(1000);
    run = false;            // 线程t不会如预想的停下来
}

  1. 初始状态， t 线程刚开始从主内存读取了 run 的值到工作内存。

  2. 因为 t 线程要频繁从主内存中读取 run 的值，JIT 编译器会将 run 的值缓存至自己工作内存中的高速缓存中，提高效率。

  3. 1 秒之后，main 线程修改了 run 的值，并同步至主存，而 t 是从工作内存中的高速缓存中读取变量的，结果永远是旧值。

5.2.2 解决方法

volatile（易变关键字）：它可以用来修饰成员变量和静态成员变量。

他可以避免线程从自己的工作缓存中查找变量的值，必须到主存中获取它的值。线程操作 volatile 变量都是直接操作主存。

5.2.3 可见性

前面例子体现的实际就是可见性，它保证的是在多个线程之间，一个线程对 volatile 变量的修改对另一个线程可见。

不能保证原子性，仅用在一个写线程，多个读线程的情况。上例从字节码理解是这样的：

xxxxxxxxxx
getstatic     run           // 线程 t 获取 run true
getstatic     run           // 线程 t 获取 run true
getstatic     run           // 线程 t 获取 run true
getstatic     run           // 线程 t 获取 run true
putstatic     run           // 线程 main 修改 run 为 false， 仅此一次
getstatic     run           // 线程 t 获取 run false

比较一下之前讲线程安全时举的例子：两个线程一个 i++ 一个 i--，只能保证看到最新值，不能解决指令交错。

xxxxxxxxxx
getstatic     i             // 线程1-获取静态变量i的值 线程内i=0
getstatic     i             // 线程2-获取静态变量i的值 线程内i=0
iconst_1                    // 线程1-准备常量1
iadd                        // 线程1-自增 线程内i=1    
putstatic     i             // 线程1-将修改后的值存入静态变量i 静态变量i=1
iconst_1                    // 线程2-准备常量1
isub                        // 线程2-自减 线程内i=-1
putstatic     i             // 线程2-将修改后的值存入静态变量i 静态变量i=-1

注意：

  • synchronized 语句块既可以保证代码块的原子性，也同时保证代码块内变量的可见性。但重量级操作，性能相对更低。

  • 如果在死循环中加入 System.out.println() 会发现即使不加 volatile，线程 t 也能正确看到对 run 变量的修改，为什么？

    因为 I/O操作会将变量的值刷新到主内存中。因此，每次打印操作都会确保 run 变量的值是最新的，并且对其他线程可见。

5.3 有序性

5.3.1 指令重排

指令重排是指在不改变程序输出的前提下，编译器或处理器可以重新安排代码的执行顺序以优化性能。

然而，在多线程环境中，这可能导致一些线程看到共享变量的不一致状态。

xxxxxxxxxx
class ReorderingExample {
    private boolean flag = false;
    private int number = 0;


    public void writer() {
        number = 42;                    // 语句1
        flag = true;                    // 语句2
    }


    public void reader() {
        while (!flag) {
            Thread.yield();             // 让出CPU时间片
        }
        System.out.println(number);     // 可能打印出0
    }
}

在这个例子中，如果指令重排发生，reader 方法可能会在 flag 被设置为 true 之前看到 number 的值，导致打印出 0。

5.3.2 有序性

有序性是指在多线程环境中，操作的执行顺序对所有线程都是一致的。

  1. 使用 volatile 关键字：可以防止指令重排，确保 volatile 变量的写操作在它之前的普通变量写操作对其他线程可见。

  2. 使用 synchronized 关键字：通过同步块或同步方法来保证代码块的执行顺序。

  3. 使用 final 关键字：可以保证变量的写操作在初始化完成之前不会被其他线程访问。

5.3.3 happens-before

happens-before 关系是 Java 内存模型中定义的一种规则，用来指定两个操作之间的内存可见性。

如果一个操作 A happens-before 操作 B，那么 A 操作的结果对 B 操作可见，且 A 的执行顺序排在 B 之前。

  • 程序的顺序规则：在一个线程内，按照程序顺序，前面的操作 happens-before 后面的操作。

  • 监视器锁规则：一个解锁操作 happens-before 后续的同一个锁的锁定操作。

  • volatile 变量规则：对一个 volatile 变量的写操作 happens-before 后续对这个变量的读操作。

  • 传递性：如果A happens-before B，且B happens-before C，那么A happens-before C。

xxxxxxxxxx
int x = 0, y = 0;
int a, b;


Thread t1 = new Thread(() -> {
    x = 1;              // 操作1
    y = x;              // 操作2
});


Thread t2 = new Thread(() -> {
    a = y;              // 操作3
    b = x;              // 操作4
});


t1.start(); t2.start();

在这个例子中，操作1 happens-before 操作2，操作2 happens-before 操作3，因此操作1 happens-before 操作3。

同样，操作1 happens-before 操作4。这意味着在 t2 线程中，a 和 b 的值应该分别是 1 和 1。

5.4 CAS 与 原子类

5.4.1 CAS

CAS（Compare-And-Swap）实现多线程环境下的原子操作。它包含三个操作数：内存位置（V）、预期原值（A）和新值（B）。

如果内存位置的值与预期原值相匹配，则处理器会自动将该位置值更新为新值。CAS 操作是原子的，它在执行时会锁定内存位置。

xxxxxxxxxx
public class CasExample {
    private AtomicInteger atomicInteger = new AtomicInteger(0);


    public void increment() {
        while (true) {
            int currentValue = atomicInteger.get();                     // 获取当前值
            int newValue = currentValue + 1;                            // 计算新值
            if (atomicInteger.compareAndSet(currentValue, newValue)) {  // 尝试更新值
                break;      // 如果成功则退出循环
            }
            // 如果失败则循环重试
        }
    }


    public static void main(String[] args) {
        CasExample example = new CasExample();
        example.increment();
        System.out.println(example.atomicInteger.get());                // 输出：1
    }
}

5.4.2 乐观锁与悲观锁

  • 乐观锁 假设在大多数情况下，数据在竞争中不会发生冲突，因此在数据访问时不会上锁。

    乐观锁通常通过在数据上附加版本号或者时间戳来实现，如果更新时与读取时的不一致，则认为数据已被修改，此时操作失败。

  • 悲观锁 假设数据在竞争中会发生冲突，因此在数据访问时会直接上锁。

    悲观锁通常通过数据库的锁机制或 Java 中的 synchronized 关键字实现。

xxxxxxxxxx
public class OptimisticLockExample {
    private AtomicStampedReference<String> atomicStampedReference 
        = new AtomicStampedReference<>("Initial Value", 0);


    public boolean attemptToUpdate(int expectedStamp, String newValue) {
        while (true) {
            String current = atomicStampedReference.getgetReference();
            int[] stampHolder = new int[1];
            if (atomicStampedReference.compareAndSet(
                current, newValue, expectedStamp, expectedStamp + 1, stampHolder)) {
                return true;                    // 更新成功
            }
            expectedStamp = stampHolder[0];     // 更新失败，获取新的时间戳
        }
    }


    public static void main(String[] args) {
        OptimisticLockExample example = new OptimisticLockExample();
        boolean result = example.attemptToUpdate(0, "New Value");
        System.out.println(result);             // 输出：true 或 false
    }
}

这些类利用 CAS 操作来保证操作的原子性。原子类示例：

  • AtomicBoolean：用于原子更新布尔值。

  • AtomicInteger：用于原子更新整数值。

  • AtomicLong：用于原子更新长整数值。

  • AtomicReference：用于原子更新对象引用。

xxxxxxxxxx
public class AtomicClassExample {

        atomicInteger.incrementAndGet();                    // 原子地增加值
    }
